# Implementation Plan: Function Argument Correction Using Serena

**Branch**: `002-serena` | **Date**: 2025-10-11 | **Spec**: [spec.md](./spec.md)
**Input**: Feature specification from `/specs/002-serena/spec.md`

## Summary

This feature systematically identifies and fixes incorrect function arguments across the solid-imager codebase using Serena's semantic code analysis tools. The primary issue is duplicate database modules (`db/db.ts` and `db/index.ts`) with conflicting function names, plus general function call validation. The solution uses Serena's `find_symbol`, `find_referencing_symbols`, and `replace_regex` tools to discover, analyze, and fix argument mismatches automatically, with manual review fallback for complex cases.

## Implementation Phases

### Phase 0: Preparation

**Duration**: 5 minutes

**Actions**:
1. Delete `src/db/db.ts`
2. Verify no imports reference it: `grep -r "from.*db/db" src/`
3. Run baseline checks: `bun run check`, `npx tsc --noEmit`

**Deliverables**:
- ✓ Duplicate file removed
- ✓ Clean baseline established

### Phase 1: Function Discovery

**Duration**: 10 minutes

**Actions**:
1. Use Serena `find_symbol` to scan:
   - `src/lib/**/*.ts` with `include_kinds: [12]`
   - `src/services/**/*.ts` with `include_kinds: [12]`
   - `src/routes/api/**/*.ts` with `include_kinds: [12]`
   - `src/db/index.ts` with `include_kinds: [12]`
2. Build `FunctionSignature` collection
3. Parse function bodies to extract parameter info

**Deliverables**:
- `FunctionSignature[]` collection (~150 items)
- Function catalog JSON export

### Phase 2: Call Site Discovery

**Duration**: 15 minutes

**Actions**:
1. For each `FunctionSignature`, call Serena `find_referencing_symbols`
2. Parse code snippets to extract arguments
3. Build `CallSite` collection

**Deliverables**:
- `CallSite[]` collection (~500-1000 items)
- Call site catalog JSON export

### Phase 3: Mismatch Detection

**Duration**: 5 minutes (in-memory)

**Actions**:
1. For each `CallSite`, compare against `FunctionSignature`
2. Check: argument count, order, types
3. Generate `ArgumentMismatch` for discrepancies
4. Classify severity (error/warning/info)
5. Calculate fix confidence

**Deliverables**:
- `ArgumentMismatch[]` collection (~10-50 items expected)
- Mismatch report (grouped by file, function, severity)

### Phase 4: Fix Application

**Duration**: 20 minutes

**Actions**:
1. Filter fixable mismatches (confidence ≥80%)
2. Group by file
3. For each file:
   a. Sort by line number (descending)
   b. Generate regex fix strategy
   c. Apply via Serena `replace_regex`
   d. Record success/failure
   e. Run `npx tsc --noEmit` on file
   f. Rollback if errors

**Deliverables**:
- `AppliedFix[]` collection (~8-15 items expected)
- `ManualReviewItem[]` collection (~2-5 items expected)
- Modified source files with fixes applied

### Phase 5: Verification

**Duration**: 10 minutes

**Actions**:
1. Run full TypeScript compilation: `npx tsc --noEmit`
2. Run Biome checks: `bun run check`
3. Run unit tests: `bun test`
4. Run build: `bun run build`
5. Compare against baseline

**Deliverables**:
- Verification report (pass/fail for each check)
- Diff against baseline errors

### Phase 6: Reporting

**Duration**: 5 minutes

**Actions**:
1. Generate `fix-report.md` (human-readable summary)
2. Generate `fix-report.json` (machine-readable data)
3. Generate `manual-review.csv` (spreadsheet for review items)
4. Update this plan with "Implementation Complete" section

**Deliverables**:
- `specs/002-serena/fix-report.md`
- `specs/002-serena/fix-report.json`
- `specs/002-serena/manual-review.csv`
