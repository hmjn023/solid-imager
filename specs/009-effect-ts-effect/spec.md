# Feature Specification: Effect.tsの導入を元に戻す

## 1. Overview
Effect.tsの試験的な導入により、エラーハンドリングとテストに関する情報不足が原因で複数の問題が発生しています。この機能仕様は、Effect.tsが導入されていない状態にプロジェクトをロールバックするための計画を概説します。

## 2. Motivation
Effect.tsの現在の実装は、プロジェクトの安定性と開発効率に悪影響を与えています。特に、以下の点が問題となっています。
- エラーハンドリングの複雑さ: Effect.tsのパラダイムに合わせた適切なエラーハンドリングのパターンが確立されておらず、デバッグが困難になっている。
- テストの困難さ: Effect.tsを使用するコードのテスト方法に関する明確なガイドラインや既存のテストが不足しており、新しい機能の追加や既存機能の変更がリスクを伴う。
これらの問題を解決し、開発プロセスを円滑に進めるために、Effect.tsの導入を元に戻す必要があります。

## 3. Goals
- Effect.tsに関連するすべての依存関係とコードをプロジェクトから削除する。
- Effect.tsの導入前の安定した状態にプロジェクトを復元する。
- エラーハンドリングとテストの明確なパターンを再確立する。

## 4. Non-Goals
- Effect.tsの代替となる新しいライブラリやフレームワークを導入すること。
- Effect.tsの導入によって発生した既存のバグを修正すること（これは別のタスクとして扱われるべきです）。

## Clarifications

### Session 2025-10-19
- Q: ロールバック後のパフォーマンス目標は？ → A: 現状維持
- Q: ロールバック後のシステム監視（ログ、メトリクス）への影響は？ → A: 影響なし
- Q: Effect.ts導入によって変更されたAPIは、外部システムとの連携に影響を与えますか？ → A: いいえ
- Q: ロールバック後に特に注意すべき既存機能のシナリオは？ → A: データの永続化
- Q: Effect.ts導入によってCI/CDパイプラインに変更がありましたか？ → A: いいえ

## 5. Requirements

### 5.1. Functional Requirements
- Effect.tsのライブラリおよび関連する型定義を`package.json`および`tsconfig.json`から削除すること。
- Effect.tsのAPIを使用しているすべてのコードを特定し、削除または既存のパターンに置き換えること。
- Effect.tsに関連する設定ファイルやスクリプトを削除すること。

### 5.2. Non-Functional Requirements
- ロールバック後も、既存の機能が正しく動作すること。
- パフォーマンスは現状維持を目標とする。
- ビルドプロセスが正常に完了し、エラーが発生しないこと。
- テストスイートが正常に実行され、すべてのテストがパスすること。

## 6. Design

### 6.1. Architecture
Effect.tsに関連するコードは、主に`src/application/services`や`src/infrastructure`ディレクトリに存在する可能性があります。これらのファイルを特定し、Effect.tsへの依存を排除します。

### 6.2. Data Model Changes
データモデルへの変更は想定されていません。

### 6.3. API Changes
Effect.tsの導入によってAPIが直接変更された場合、それらを元の状態に戻す必要があります。

### 6.4. UI/UX Changes
UI/UXへの変更は想定されていません。

## 7. Test Cases
- Effect.tsの依存関係が`package.json`から削除されていることを確認する。
- `bun install`が正常に実行されることを確認する。
- `bun run dev`が正常に起動し、アプリケーションが動作することを確認する。
- 既存のユニットテストおよび統合テストがすべてパスすることを確認する。
- アプリケーションの主要な機能が手動テストで正常に動作することを確認する。
- 特にデータの永続化（DBへの保存、取得、更新、削除など）が正常に動作することを確認する。

## 8. Rollback Plan
この変更自体がロールバックであるため、追加のロールバック計画は不要です。

## 9. Open Questions
- Effect.tsの導入によって影響を受けた可能性のある、まだ特定されていない領域はありますか？
- Effect.tsの導入によって追加された、または変更された可能性のあるCI/CDパイプラインのステップはありますか？

## 10. Future Considerations
- 今後のライブラリ導入時には、より厳格な評価プロセスと、エラーハンドリングおよびテスト戦略に関する十分な情報収集を行うこと。
