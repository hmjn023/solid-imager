# Research: Effect.tsの導入を元に戻す

## 1. Effect.tsのコードベースからの特定と削除

### 1.1. `package.json`と`bun.lock`からの依存関係の削除
Effect.tsに関連するパッケージを`package.json`の`dependencies`および`devDependencies`から特定し、削除します。その後、`bun install`を実行して`bun.lock`ファイルを更新し、依存関係をクリーンアップします。

### 1.2. `tsconfig.json`からの型定義の削除
Effect.tsに関連する型定義が`tsconfig.json`の`types`または`typeRoots`に含まれている場合、それらを削除します。

### 1.3. コードベースからのEffect.tsのインポートと使用箇所の特定
プロジェクト全体で`Effect`または`@effect`をインポートしている箇所を検索します。これらの箇所は、Effect.tsのAPIを使用している可能性が高いです。検索には`grep -r "import.*Effect" src/`のようなコマンドが有効です。

### 1.4. Effect.tsのAPIを使用しているコードの置き換えまたは削除
特定されたEffect.tsのAPIを使用しているコードを、既存の非Effect.tsのパターン（例: Promise、try-catch、またはSolid.jsのリアクティブプリミティブ）に置き換えるか、完全に削除します。特に、エラーハンドリングロジックは慎重にレビューし、適切な代替手段を導入する必要があります。

### 1.5. Effect.tsに関連する設定ファイルやスクリプトの削除
Effect.ts専用の設定ファイル（例: `effect.config.ts`のようなもの）や、Effect.tsのビルド/テスト/実行に関連するカスタムスクリプトがあれば、それらを削除します。

## 2. テスト戦略の再確立

### 2.1. Effect.tsに依存するテストの修正または削除
Effect.tsのAPIを使用しているテストファイルやテストケースを特定し、Effect.tsへの依存を排除するように修正するか、関連するコードが削除された場合はテスト自体を削除します。

### 2.2. データの永続化に関するテストの強化
ユーザーからのフィードバックに基づき、データの永続化（DBへの保存、取得、更新、削除など）に関する既存のテストをレビューし、必要に応じて強化します。Effect.tsの導入によって影響を受けた可能性のあるDB操作ロジックが、ロールバック後も正しく機能することを確認するためのテストを追加します。

## 3. ロールバック後の検証

### 3.1. ビルドと実行の確認
`bun install`、`bun run dev`、`bun run build`、`bun run start`がすべてエラーなく実行され、アプリケーションが正常に動作することを確認します。

### 3.2. 既存機能の動作確認
主要な機能（特にデータの永続化）が手動テストで正常に動作することを確認します。

### 3.3. テストスイートの実行
`bun run test`を実行し、すべてのテストがパスすることを確認します。

## 4. 影響範囲の再確認

### 4.1. APIの変更
Effect.tsの導入によってAPIが直接変更された場合、それらが元の状態に戻っていることを確認します。外部システムとの連携に影響がないことを再確認します。

### 4.2. CI/CDパイプライン
CI/CDパイプラインに変更がないことを再確認します。ビルド、テスト、デプロイの各ステップが正常に機能することを確認します。
