# Implementation Plan: Reorganize src/lib and src/utils Architecture

**Branch**: `005-src-lib-utils` | **Date**: 2025-10-11 | **Spec**: [spec.md](./spec.md)
**Input**: Feature specification from `/specs/005-src-lib-utils/spec.md`

## Summary

This feature reorganizes the `src/lib/` and `src/utils/` directories to follow modern Clean Architecture / Hexagonal Architecture patterns. The primary goal is to establish clear separation of concerns by organizing code into domain, application, infrastructure, and presentation layers. This will improve code maintainability, testability, and developer navigation while eliminating code duplication.

**Technical Approach**: Use Serena MCP tools for precise file operations and import tracking. Perform incremental migration by layer, updating imports at each step, and validating with existing tests to ensure zero breakage.

## Phase 2: Implementation Strategy

### Migration Approach

The implementation follows a **layered, incremental migration** strategy to ensure zero downtime and continuous validation:

1. **Create new directory structure** (empty directories)
2. **Migrate layer by layer** in this order:
   - Domain layer (pure functions, no dependencies)
   - Infrastructure layer (I/O, drivers, API clients)
   - Application layer (services - move not modify)
   - Presentation layer (minimal changes)
3. **Update imports** after each major module migration
4. **Run tests** after each layer to catch issues early
5. **Remove old directories** only after all imports updated

### Key Migration Rules

1. **Use Serena tools exclusively** for file operations:
   - `find_symbol` and `find_referencing_symbols` to track usage
   - `create_text_file` for new files
   - `replace_regex` for efficient import updates
   - `execute_shell_command` for file moves when appropriate

2. **Split mixed-concern files** before moving:
   - `src/lib/api/media.ts` has business logic в†’ extract to domain
   - `src/lib/helpers/storage-drivers.ts` has multiple drivers в†’ split to individual files
   - `src/lib/types.ts` and `src/lib/schemas.ts` в†’ split by domain

3. **Maintain test coverage**:
   - Update test imports immediately after moving source files
   - Run affected tests after each module migration
   - No test logic changes (only import paths)

4. **Handle re-exports carefully**:
   - Some files re-export from other modules
   - Update re-export paths or eliminate unnecessary re-exports
   - Prefer direct imports in new structure

### Critical Files Requiring Special Handling

1. **`src/lib/api/media.ts`** (~250 lines):
   - Contains API client functions + business logic
   - Must split business logic в†’ `domain/media/` or `application/services/media-service.ts`
   - Keep API client functions в†’ `infrastructure/api-clients/media.ts`

2. **`src/lib/types.ts`** (~28 type definitions):
   - Split by domain: media, sources, tags, categories, characters, ips
   - Keep cross-domain types in `shared/types/`

3. **`src/lib/schemas.ts`** (~7 validation schemas):
   - Split by domain matching types organization
   - Update schema imports in routes and services

4. **`src/lib/helpers/`** (5 files):
   - Distribute to appropriate layers based on responsibility
   - Eliminate the helpers directory entirely

5. **`src/services/thumbnail-jobs.ts`**:
   - Move to `infrastructure/jobs/` (background processing)

### Import Update Strategy

**Estimated ~150+ import statements to update across:**
- ~40 route files in `src/routes/`
- ~19 service files in `src/services/`
- ~10 component files in `src/components/`
- ~80+ test files in `src/tests/`

**Approach**:
1. Use `find_referencing_symbols` to identify all imports before moving files
2. Update imports using `replace_regex` with careful matching
3. Prefer project-wide import path updates after each layer completes
4. Use TypeScript compiler to catch any missed imports

### Validation Gates

Each migration phase must pass these gates before proceeding:

1. **After creating new directories**: Build succeeds (no structural issues)
2. **After migrating domain layer**: All unit tests pass, no import errors
3. **After migrating infrastructure layer**: Integration tests pass
4. **After migrating application layer**: All service tests pass
5. **After migrating presentation layer**: E2E tests pass
6. **Final validation**: Full test suite passes, build succeeds, no linter errors

### Rollback Strategy

If any migration phase fails validation:
1. **Revert the last commit** (each layer should be one commit)
2. **Fix the identified issue** in isolation
3. **Re-attempt the migration** for that layer
4. **Do not proceed** to next layer until current layer passes all gates

## Phase 3: Testing Strategy

### Test Updates Required

All existing tests remain unchanged in logic, only import paths update:

1. **Unit Tests** (`src/tests/unit/`):
   - Update imports for domain and infrastructure modules
   - No test logic changes required

2. **Integration Tests** (`src/tests/integration/`):
   - Update service imports
   - No test logic changes required

3. **E2E Tests** (`src/tests/e2e/`):
   - No changes required (tests use public APIs)

4. **API Tests** (`src/tests/api/`):
   - No changes required (tests use route handlers)

### New Tests

No new tests required - this is pure refactoring. All existing functionality and test coverage is preserved.

### Test Execution Schedule

- **After domain layer migration**: Run unit tests
- **After infrastructure layer migration**: Run integration tests
- **After application layer migration**: Run all tests except e2e
- **After presentation layer migration**: Run full test suite including e2e
- **Final validation**: Run `bun run test` (full suite) + `bun run check` (type check)
