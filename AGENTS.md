# GEMINI.md - solid-imager

## Project Overview

This project is a comprehensive media management system specifically designed for assets (images, videos, etc.) generated by ComfyUI. It provides a backend API and a web frontend to organize, search, and serve these media files.

**Key Features:**
- **Media Source Management:** Manages files from local directories, SFTP, and AWS S3.
- **Media Delivery & Thumbnail Generation:** Serves media files and generates thumbnails.
- **Metadata Extraction:** Extracts and stores crucial metadata from generated files, such as prompts and workflows.
- **Real-time Updates:** Uses Server-Sent Events (SSE) for live updates on file changes.
- **Media Upload & Editing:** Supports uploading new files and editing existing media information.
- **Search & Organization:** Provides powerful tagging, searching, sorting, character/IP management, and categorization features.
- **Configuration Management:** Allows for managing system settings.

**Terminology Definition:**
- **Media Source**: The location where files are stored and their management method.
- **Subdirectory**: A directory within a specific media source, and its recursive subdirectories.
- **Media List**: A screen displaying media and its subdirectories within a specific media source in a grid format.
- **Media Information**: Information such as name, media dimensions, file size, creation date, and update date.
- **Generation Information**: Information stored in the PNG tEXt chunk when media is generated (prompt, workflow).
- **Tags**: Keyword information entered into the workflow when generating media (included in prompt and workflow).

## System Requirements

### Supported Media Sources
- Local directories on the machine where the backend is running.
- SFTP connection directories.
- AWS S3.

### Supported Media Formats
**Initial Support:**
- Image files: `png`, `jpg`, `jpeg`, `webp`, `gif`

**Future Support Planned:**
- Video files: `mp4`, `webm`, `mov` + ffmpeg thumbnail generation.
- Audio files: `mp3`, `wav`, `flac` + waveform generation.

## Development Conventions

### Database

The database schema is managed with Drizzle ORM and is defined in `src/db/schema.ts`.

- **Generating Migrations:** After making changes to the schema in `src/db/schema.ts`, generate a new migration file:
  ```bash
  bun run db:generate
  ```

- **Applying Migrations:** To apply pending migrations to the database:
  ```bash
  bun run db:migrate
  ```

- **Database Studio:** To browse the database contents, use Drizzle Studio:
  ```bash
  bun run db:studio
  ```

### Code Quality

The project uses **Biome** for linting and formatting. Before committing any changes, it is highly recommended to run the `check` script to ensure code consistency.

- **Check and Fix:** Lint, format, and apply safe fixes.
  ```bash
  bun run check
  ```

### Code Style

- **Component Naming:** Components should be named using `PascalCase` (e.g., `MediaGrid.tsx`).
- **Type Imports:** When importing only types or interfaces, always use `import type`.
  ```typescript
  import type { Media } from '~/db/schema';
  ```
- **Styling:** All styling should be done using Tailwind CSS classes directly in the JSX.
- **Import Alias:** The project is configured with a `~/*` alias for the `src/*` directory. Use this for cleaner import paths.
  ```typescript
  import Nav from '~/components/Nav';
  ```

### API Route Conventions

- **JSON Response:** For API routes returning simple JavaScript objects, it is generally not necessary to explicitly import and use `json()` from `solid-start/server`. SolidStart automatically serializes plain JavaScript objects returned from `GET` (and other HTTP method) functions into JSON responses.
- **Non-JSON Responses:** For responses involving non-JSON data types such as images or binary data, the appropriate handling (e.g., setting `Content-Type` headers and returning raw data) is currently unverified and may require explicit handling.

### Bun API Usage

- **Avoid Bun-specific APIs:** To ensure better portability and avoid vendor lock-in, refrain from using Bun's specific APIs (e.g., `Bun.file()`, `Bun.serve()`) unless absolutely necessary and explicitly approved. Prefer Node.js compatible APIs or standard Web APIs where possible.

```sql
  id UUID PRIMARY KEY,
```

## API Design

### Common Principles
- Adhere to RESTful design principles.
- Consistent endpoint naming conventions.
- Appropriate use of HTTP verbs.
- Unified error handling.
- **Path Wildcards (`*`)**: In paths like `/api/sources/:id/media/*` or `/api/sources/:id/directories/*`, `*` represents the "complete relative path" of media or directories within the source. In SolidStart routing, this is expressed using `[...param]`.
- **Path Processing Notes**: `?path=subdir` for media list acquisition and `*` wildcards for specific media operations correspond to different use cases (filtering and specifying specific resources, respectively).
- **API Versioning**: Consider introducing API versioning (e.g., `/v1/api/...`) for future scalability and backward compatibility.

## Feature Details Specification

### 2. Media Delivery and Thumbnail Creation Function

#### Thumbnail Specifications
- Size: Specified by query parameter (?size=200, ?size=400, etc.)
- Cache: Stored in local disk cache.
- Supported Formats: PNG, JPEG, WebP.
- Generation: Background generation (job queue).

#### Thumbnail Generation Timing
- When adding a media source: Start batch generation of existing media.
- When new media is detected by SSE: Start individual media generation.
- Manual trigger: Provide a regeneration API.

#### Progress Notification (SSE)
```typescript
interface ThumbnailProgress {
  type: 'thumbnail_progress';
  sourceId: string;
  status: 'started' | 'processing' | 'completed' | 'error';
  progress: {
    current: number;    // Number of media processed
    total: number;      // Total number of media
    currentFile?: string;
  };
  error?: string;
}
```

### 3. Media Metadata Extraction Function

#### Data Structure
```typescript
interface MediaMetadata {
  // ComfyUI standard fields (JSON parsing)
  prompt?: object;     // JSON.parse(textChunk["prompt"])
  workflow?: object;   // JSON.parse(textChunk["workflow"])
  
  // Other keys (as text)
  parameters?: string;
  [key: string]: any;  // For future expansion
}
```

#### Processing Specifications
1. Read tEXt chunks from PNG media.
2. Attempt JSON parsing for `prompt` and `workflow` keys.
3. If parsing fails, retain as text.
4. Other keys remain as text.
5. If no metadata exists, return an empty object.

### 4. SSE Function

#### Scope
- Only supports `type: 'local'` media sources.
- SFTP/S3 not supported.

#### Data Structure
```typescript
interface FileSystemEvent {
  type: 'added' | 'deleted' | 'modified';
  sourceId: string;
  filePath: string;    // Relative path
  timestamp: Date;
}
```

#### Monitoring Specifications
- Library: `chokidar` for file system monitoring.
- Target: Media files (.png, .jpg, .jpeg, .webp, etc.).
- Scope: Recursively monitors subdirectories.
- Error: Non-local sources will return "Only local file sources supported" error.

### 5. Media Upload Function

#### Request Format (multipart/form-data)

```typescript
interface UploadRequest {

```

#### Response Format
```typescript
interface UploadResponse {

  filePath: string;              // Saved path
  conflict?: {                   // On conflict

    suggestedName: string;
  }

```

#### Support Status
- **Phase 1**: Only `type: 'local'` supported.
- **Phase 2**: SFTP/S3 support planned.

#### File Name Processing
- Custom name takes precedence; if not specified, original file name is used.
- If `autoIncrement: true`, uses `media_001.png`, `media_002.png` format.
- On conflict, returns `conflict` information and requests user confirmation.
