# GEMINI.md - solid-imager

## Project Overview

This project is a comprehensive media management system specifically designed for assets (images, videos, etc.) generated by ComfyUI. It provides a backend API and a web frontend to organize, search, and serve these media files.

**Key Features:**
- **Media Source Management:** Register, update, and delete media sources (local, SFTP, AWS S3). Test connections and view status.
- **Media Delivery & Thumbnail Generation:** Serve media thumbnails with specified sizes, generate thumbnails in the background, and manage thumbnail cache.
- **Metadata Extraction:** Extract and store detailed metadata from generated files, including prompts, workflows, model names, seed values, CFG scales, and steps.
- **Real-time Updates:** Use Server-Sent Events (SSE) to notify clients of file system changes (additions, deletions, modifications) and thumbnail generation progress.
- **Media Upload & Editing:** Upload new media with custom filenames, auto-incrementing, duplicate checking, and overwrite options. Edit media information such as filename, description, source URL, and tags.
- **Search & Organization:** Search and sort media by filename, tags, date range, creation date, update date, and file size. Manage categories, IPs, characters, projects, and collections.
- **Configuration Management:** Retrieve, update, and reset application settings stored in `config.json`.

## System Requirements

The system supports various media sources (local, SFTP, S3) and formats (images, videos, audio). For a detailed breakdown of all requirements, please refer to [docs/design/02-system-requirements.md](docs/design/02-system-requirements.md).

## Development Conventions

### Architecture
This project adopts a **Clean Architecture / Hexagonal Architecture** approach with four primary layers:
1. **Domain Layer** (`src/domain/`) - Business logic, domain models, pure functions.
2. **Application Layer** (`src/application/`) - Use case orchestration, service layer.
3. **Infrastructure Layer** (`src/infrastructure/`) - External integrations, I/O, adapters.
4. **Presentation Layer** (`src/presentation/`) - UI utilities.

**Dependency Rules:**
```
Presentation → Application → Domain ← Infrastructure
```
- Domain layer has zero external dependencies.
- Infrastructure depends only on domain (for types and interfaces).
- Application orchestrates domain and infrastructure.
- Presentation can depend on any layer.

For more details, refer to `docs/architecture/ADR-001-clean-architecture.md` and `docs/architecture/ARCHITECTURE.md`.

### Database

The database schema is managed with Drizzle ORM and is defined in `src/db/schema.ts`.

- **Generating Migrations:** After making changes to the schema in `src/db/schema.ts`, generate a new migration file:
  ```bash
  bun run db:generate
  ```

- **Applying Migrations:** To apply pending migrations to the database:
  ```bash
  bun run db:migrate
  ```

- **Database Studio:** To browse the database contents, use Drizzle Studio:
  ```bash
  bun run db:studio
  ```

### Code Quality

The project uses **Biome** for linting and formatting. Before committing any changes, it is highly recommended to run the `check` script to ensure code consistency.

- **Check and Fix:** Lint, format, and apply safe fixes.
  ```bash
  bun run check
  ```

### Code Style

- **Component Naming:** Components should be named using `PascalCase` (e.g., `MediaGrid.tsx`).
- **Type Imports:** When importing only types or interfaces, always use `import type`.
  ```typescript
  import type { Media } from '~/db/schema';
  ```
- **Styling:** All styling should be done using Tailwind CSS classes directly in the JSX.
- **Import Alias:** The project is configured with a `~/*` alias for the `src/*` directory. Use this for cleaner import paths.
  ```typescript
  import Nav from '~/components/Nav';
  ```

### API Route Conventions

- **JSON Response:** For API routes returning simple JavaScript objects, it is generally not necessary to explicitly import and use `json()` from `solid-start/server`. SolidStart automatically serializes plain JavaScript objects returned from `GET` (and other HTTP method) functions into JSON responses.
- **Non-JSON Responses:** For responses involving non-JSON data types such as images or binary data, the appropriate handling (e.g., setting `Content-Type` headers and returning raw data) is currently unverified and may require explicit handling.

### Bun API Usage

- **Avoid Bun-specific APIs:** To ensure better portability and avoid vendor lock-in, refrain from using Bun's specific APIs (e.g., `Bun.file()`, `Bun.serve()`) unless absolutely necessary and explicitly approved. Prefer Node.js compatible APIs or standard Web APIs where possible.
