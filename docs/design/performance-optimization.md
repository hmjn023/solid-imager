# パフォーマンス改善計画 (Performance Optimization Plan)

このドキュメントでは、現在のコードベースにおけるパフォーマンスとリソース消費の改善点（ボトルネック）と、それに対する具体的な改善案をまとめています。

## 1. データベース・検索クエリ (Database & Queries)

### 1-1. 動的フィルタリングクエリの非効率性
- **現状**: `media-repository-utils.ts` において、`tags`, `projects`, `ips` などの複数条件でフィルタリングする際、`WHERE id IN (SELECT ...)` 形式のサブクエリを多用している。
- **問題点**: フィルタ条件が増えるほどサブクエリがネストし、処理負荷が指数関数的に増大する可能性がある（実行計画が最適化されにくい）。
- **改善案**:
    - `EXISTS` 句を使用したクエリへの書き換え。
    - 結合 (`JOIN`) と `GROUP BY` を適切に使用したクエリ構成への見直し。

### 1-2. 文字列検索（LIKE）のパフォーマンス
- **現状**: ファイル名や説明文の検索に中間一致 (`LIKE %query%`) を使用している。
- **問題点**: 標準のB-Treeインデックスが使用できず、テーブルフルスキャンが発生する。データ量が増えると検索速度が著しく低下する。
- **改善案**:
    - PostgreSQLの **pg_trgm (トリグラム)** インデックスの導入。
    - 全文検索 (Full Text Search - `tsvector`) への移行。

### 1-3. インデックスの不足 (Missing Indexes)
- **現状**: `media_tags` などの中間テーブルにおいて、逆引き（`tag_id` から `media_id` を探すなど）のためのインデックスが明示されていない。
- **改善案**:
    - `media_tags(tag_id)`
    - `media_authors(author_id)`
    - `media_projects(project_id)`
    - `media_characters(character_id)`
    - `media_ips(ip_id)`
    などの外部キー列にインデックスを追加する。

## 2. バックエンド処理・リソース効率 (Backend Resource Efficiency)

### 2-1. メモリ使用量の最適化（バックアップ処理）
- **現状**: `BackupService.createDump` で、対象ソース内の全メディア・全関連データを一度にメモリ上にロードしている。
- **問題点**: メディア数が多い場合（数万件〜）、Node.jsプロセスのメモリ制限を超え、**OOM (Out Of Memory)** エラーを引き起こすリスクが高い。
- **改善案**:
    - **ストリーム処理 (Streaming)**: データベースからデータをストリームとして読み出し、一件ずつ処理してZipストリームに流す。
    - **チャンク処理 (Chunking)**: カーソルベースのページネーションを使用し、一定数（例: 1000件）ずつ処理を行う。

### 2-2. ファイル圧縮の効率化
- **現状**: 各メディアファイルを一度バッファ（メモリ）に読み込んでから `archiver` に渡している。
- **改善案**: ファイルシステムからの読み込みストリーム (`createReadStream`) を直接アーカイバにパイプし、メモリへのバッファリングを最小限に抑える。

### 2-3. N+1 Insert問題（リストア処理）
- **現状**: リストア時 (`_copyMediaMetadata` 等) に、関連データをループ内で1件ずつ `INSERT` している。
- **問題点**: データベースとのラウンドトリップ回数が膨大になり、リストア処理に非常に時間がかかる。
- **改善案**:
    - データを配列に蓄積し、`db.insert().values([...])` による **バルクインサート (Bulk Insert)** に切り替える。

### 2-4. ジョブ実行の非同期化
- **現状**: `uploadMedia` メソッド内で、ジョブのキューイング後に `startJobQueue` を呼び出し、同期的（または同じリクエストコンテキスト内）に処理を開始している。
- **改善案**:
    - ジョブの実行をリクエスト処理から完全に切り離す。
    - 独立したワーカープロセス、または常駐するジョブランナーがキューを監視・処理するアーキテクチャに変更する。

## 3. フロントエンド (Frontend & UI)

### 3-1. リストレンダリングの仮想化 (Virtualization)
- **現状**: 無限スクロールでデータを追加読み込みする際、取得したすべてのDOM要素を描画し続けている。
- **問題点**: スクロールが進むにつれてDOMノード数が増大し、ブラウザのメモリ消費量増加と描画パフォーマンスの低下（カクつき）を招く。
- **改善案**:
    - **仮想スクロール (Virtual Scrolling)** の導入（例: `@tanstack/solid-virtual`）。
    - ビューポート（画面内）に見えている要素のみをレンダリングし、画面外の要素をDOMから削除またはリサイクルする。
