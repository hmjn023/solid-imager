# コード削減と共通化に関する調査レポート

本レポートでは、プロジェクト内のコードベースを調査し、共通化や書き直しによってコード量を削減し、保守性を向上させるための提案をまとめます。

## 1. 現状の課題概要

調査の結果、主にバックエンド（リポジトリ、サービス、ルーター）において、標準的なCRUD操作の実装が高度に重複していることが確認されました。また、フロントエンドにおいても、APIクライアントの定義や管理画面の実装に冗長性が見られます。

これらの重複は、将来的なスキーマ変更や機能追加の際に、多数のファイルを修正する必要性を生じさせ、バグの温床となる可能性があります。

## 2. バックエンドの削減案

### 2.1 汎用リポジトリ (Generic Repository) の導入

現在、`TagRepository`, `CategoryRepository`, `ProjectRepository` などのリポジトリは、以下のような標準的なメソッドを個別に実装しています。

*   `findAll()`
*   `findById(id)`
*   `create(data)`
*   `update(id, data)`
*   `delete(id)`

これらの実装は、対象となるテーブルとスキーマが異なるだけで、ロジック自体（Drizzle ORMの呼び出し、エラーハンドリング、トランザクション処理）はほぼ同一です。

**提案:**
`DrizzleRepository<T>` のような汎用クラス、またはファクトリ関数を作成し、標準的なCRUD操作を共通化することを推奨します。

**削減効果:**
各リポジトリファイルのコード量を **60-80%** 削減可能です。

### 2.2 サービス層の簡素化

多くのサービス（例: `ProjectService`）は、リポジトリのメソッドをそのまま呼び出すだけの「パススルー」になっています。ビジネスロジックが存在しない単純なCRUD操作において、この層は冗長です。

**提案:**
*   単純なCRUDについては、ルーターから直接リポジトリ（または汎用リポジトリ）を呼び出す構成に変更する。
*   あるいは、`CrudService<T>` のような汎用サービスを導入する。

**削減効果:**
多数の「薄い」サービスファイルを削除、または数行の定義に圧縮できます。

### 2.3 Router Factoryの導入 (oRPC)

`src/infrastructure/api/routers/` 内のルーター定義（`tags-router.ts` 等）は、`list`, `get`, `create`, `update`, `delete` のハンドラーを個別に定義しています。これらもパターン化されています。

**提案:**
`createCrudRouter(serviceOrRepo, schema)` のようなファクトリ関数を作成し、標準的なCRUDエンドポイントを自動生成するようにします。

**削減効果:**
ルーター定義ファイルを **数行** に短縮できます。

## 3. フロントエンドの削減案

### 3.1 APIクライアントのラッパー削除

`src/infrastructure/api-clients/` 下にある `projects-api.ts` や `tags-api.ts` は、単に `orpc` クライアントをラップしているだけです。`orpc` は既に型安全なクライアントを提供しているため、これらのラッパーは技術的に不要です。

**提案:**
コンポーネントから直接 `orpc.projects.list()` のように呼び出すか、あるいは `orpc` クライアントを拡張して使用し、手動のラッパーファイルを削除します。

**削減効果:**
`src/infrastructure/api-clients/` 内の多数のファイルを削除できます。

### 3.2 管理画面 (Manager Page) の共通コンポーネント化

`src/routes/manager.tsx` では、Projects, IPs, Characters の3つのエンティティに対して、表示・作成・編集・削除のロジックを `if/else` や `switch` 文で分岐して実装しています。

**提案:**
「一覧表示 + 作成/編集モーダル + 削除確認」の機能をセットにした `GenericCrudManager<T>` コンポーネント（または `CrudTable`）を作成し、エンティティごとの設定（カラム定義、API呼び出し関数）を渡す形にリファクタリングすることを推奨します。

**削減効果:**
`manager.tsx` の複雑な条件分岐を解消し、新しいエンティティ追加時の工数を大幅に削減できます。

## 4. 推定される削減効果まとめ

| 領域 | 現状 | 改善後 | 推定削減行数 |
| :--- | :--- | :--- | :--- |
| **Repositories** | 各ファイル 100-200行 | 各ファイル 20-50行 | -1000行以上 |
| **Services** | 各ファイル 50-100行 | 不要 (または10行) | -500行以上 |
| **Routers** | 各ファイル 50-80行 | 各ファイル 10-20行 | -300行以上 |
| **API Clients** | 多数のラッパーファイル | 0ファイル (直接利用) | -300行以上 |
| **Manager UI** | 複雑な分岐ロジック | 定義ベースの構成 | -100行以上 |

## 5. 推奨ステップ

1.  **バックエンド共通基盤の作成**:
    *   `GenericRepository` クラスの実装 (Drizzle対応)。
    *   `createCrudRouter` ファクトリの実装。
2.  **小規模なエンティティでの適用**:
    *   `Tag` や `Category` など、依存関係の少ないエンティティで上記基盤を適用し、動作検証を行う。
3.  **APIクライアントの削除**:
    *   フロントエンドでの `orpc` 直接利用への置き換えを進める。
4.  **UIの共通化**:
    *   `manager.tsx` をリファクタリングし、汎用CRUDコンポーネントを切り出す。

以上の施策により、コードベースの大幅なスリム化と、開発速度の向上が見込まれます。
