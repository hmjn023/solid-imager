# AI処理ジョブの並列数分離に関する仕様

## 背景
AIタグ付けサービス（Pythonバックエンド）はシングルスレッドで動作しており、計算コストが高いため、クライアント（Node.js/Bun）から複数のリクエストを並列に送信すると、サーバー側でキュー待ちが発生します。これにより、クライアント側のタイムアウト（デフォルト120秒）が発生しやすくなっていました。

この問題を解決するため、AI処理を伴うジョブ（タグ付けなど）と、その他のジョブ（ダウンロード、メタデータ抽出など）で並列数制限を分離します。

## 設計方針

### 1. 設定の分離
`config.json` に AIジョブ専用の並列数設定を追加します。

```json
{
  "jobs": {
    "concurrency": 3,      // 既存: 全体の並列数上限（または非AIジョブの並列数）
    "aiConcurrency": 1     // 新規: AIジョブの並列数上限
  }
}
```

### 2. ジョブタイプの分離
`processMedia` ジョブ内で行われていた AIタグ付け処理を、独立した `auto_tagging` ジョブとして分離します。

- **変更前**: `processMedia` ジョブが [メタデータ抽出] -> [サムネイル生成] -> [AIタグ付け] を一括で実行。
- **変更後**: `processMedia` ジョブは [メタデータ抽出] -> [サムネイル生成] -> [AIジョブ作成] までを行い、AIタグ付けの実処理は `auto_tagging` ジョブが行う。

### 3. Concurrency制御 (JobWorker)
JobWorkerのポーリングロジックを以下のように改修します。

- 実行中のジョブを「AI系」と「その他」でカウントする。
- **AI系ジョブ**: `activeAiJobs < aiConcurrency` の場合のみフェッチ・実行する。
- **その他ジョブ**: `activeOtherJobs < concurrency` の場合のみフェッチ・実行する。

#### AI系ジョブの定義
以下の `job.type` をAI系として扱います。
- `auto_tagging`
- `bulk_tagging_dispatch` (ディスパッチャ自体は軽いですが、フロー制御のためAI系に含めるか、あるいは除外するか検討。現状はAIリソースを使わないので除外可能だが、タグ付けフローの一部として管理する) -> **結論**: `auto_tagging` のみを対象とします。`bulk_tagging_dispatch` はジョブ生成のみで重くないため、通常の `concurrency`枠で実行します。

## データベース変更
なし（既存の `jobs` テーブル構造を使用）。

## 想定される動作
1. ユーザーが大量の画像をインポート。
2. `processMedia` ジョブが `concurrency` (3) で並列実行され、高速にサムネイル生成とメタデータ抽出が完了する。
3. `auto_tagging` ジョブがキューに積まれる。
4. `auto_tagging` ジョブは `aiConcurrency` (1) で1つずつ実行される。
5. タイムアウトのリスクが大幅に低減される。
